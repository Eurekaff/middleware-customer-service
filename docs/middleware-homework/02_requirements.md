# 需求分析

## 1. 用户角色

本项目面向课堂演示，设置三个简化角色。

### 1.1 普通用户

普通用户通过聊天页面输入问题，查看系统处理状态和最终客服回复。用户不需要登录，系统通过会话记录区分不同聊天上下文。

### 1.2 演示管理员

演示管理员通过后台演示页查看任务列表、任务状态和 MCP 工具调用日志，用于说明 Redis、Worker、MCP Server 与数据库之间的协作过程。

### 1.3 系统 Worker

Worker 不是人工用户，而是后台异步处理进程。它从 Redis List 中消费任务，调用 MCP 工具，更新数据库和 Redis 中的任务状态。

## 2. 功能需求

### 2.1 会话管理

- 创建客服会话；
- 查询会话列表；
- 查询某个会话详情；
- 会话详情包含该会话下的聊天消息。

### 2.2 消息处理

- 用户可以在前端输入问题；
- 后端保存用户消息；
- 后端创建客服处理任务；
- 后端将任务写入 Redis List；
- 后端立即返回 `task_id`；
- 前端根据 `task_id` 轮询任务状态。

### 2.3 异步任务处理

- Worker 使用阻塞方式从 Redis List 中读取任务；
- Worker 将任务状态从 `PENDING` 更新为 `PROCESSING`；
- Worker 调用 MCP 工具完成处理；
- Worker 保存机器人回复；
- Worker 保存工具调用日志；
- Worker 将任务状态更新为 `SUCCESS`、`FAILED` 或 `TRANSFERRED`。

### 2.4 MCP 工具能力

MCP Server 至少提供以下工具：

- `classify_question`：对用户问题进行分类；
- `search_knowledge_base`：检索本地知识库；
- `generate_reply`：根据分类和知识库结果生成客服回复；
- `create_ticket`：为需要后续处理的问题创建工单；
- `transfer_to_human`：判断是否需要转人工处理。

### 2.5 后台演示

- 查看任务列表；
- 查看任务状态；
- 查看 MCP 工具调用日志；
- 观察问题分类、知识库命中、回复结果和转人工结果。

## 3. 非功能需求

### 3.1 可运行性

项目应优先保证本地可运行。Redis 可通过本地安装或 `docker-compose.yml` 启动，后端、Worker、MCP Server 和前端使用普通命令行启动。

### 3.2 可演示性

页面和接口应能清楚展示完整链路，包括用户提问、任务创建、Redis 入队、Worker 消费、MCP 工具调用、数据库保存和前端展示结果。

### 3.3 可维护性

代码目录应按前端、后端、Worker、MCP Server 分开，命名清晰，核心逻辑保留必要注释。MCP 工具内部使用规则和模板实现，但结构应方便后续替换为真实大模型调用。

### 3.4 数据一致性

SQLite 负责持久化保存完整记录，Redis 负责缓存任务状态和会话最近状态。任务状态更新时需要同时更新数据库和 Redis，避免前端展示与持久化结果长期不一致。

## 4. 本次大作业边界

本项目只实现智能客服工作流的核心演示链路，不实现完整商业客服系统。主要边界如下：

- 不实现用户注册、登录和复杂权限管理；
- 不实现真实在线人工客服坐席系统；
- 不接入真实支付、订单、物流等业务系统；
- 不强制接入真实大模型 API；
- 不实现复杂统计报表；
- 不实现分布式部署和高可用治理。

## 5. 不做的复杂功能

为保证课程项目可控，明确不引入以下复杂内容：

- Kafka、RabbitMQ 等额外消息中间件；
- Kubernetes、Nginx、微服务网关；
- OAuth、RBAC 等复杂权限系统；
- 向量数据库和复杂语义检索；
- 多租户客服系统；
- 实时 WebSocket 推送；
- 复杂 SLA、工单派单和客服排班。

这些能力可以作为后续扩展方向，但不属于本次大作业的实现范围。
