# 《基于 MCP 协议与 Redis 的智能客服工作流系统设计与实现》项目概述

## 1. 项目名称

本项目名称为“基于 MCP 协议与 Redis 的智能客服工作流系统设计与实现”。项目面向中间件技术课程大作业，目标是实现一个可在本地运行、可课堂演示的智能客服工作流系统。

## 2. 课程大作业背景

中间件技术课程强调系统组件之间的解耦、异步通信、状态缓存、数据持久化以及服务协作。本项目不追求完整商业客服系统的复杂业务能力，而是通过一个简化但链路完整的 Web 系统，展示 Redis、Worker、MCP Server、SQLite 与前后端分离应用如何协同工作。

系统的核心链路为：用户在前端提交问题，后端创建客服处理任务并写入 Redis 队列，Worker 异步消费任务并调用 MCP 工具完成问题分类、知识库检索、回复生成或工单创建，最后将结果保存到 SQLite 并返回给前端展示。

## 3. 为什么选择智能客服

智能客服场景天然包含“用户请求、任务排队、异步处理、工具调用、状态追踪、日志记录”等环节，适合展示中间件技术的作用。相比普通 CRUD 系统，客服工作流更容易体现：

- 用户请求与后台处理解耦；
- 任务从创建到完成的状态流转；
- Redis 队列在异步任务处理中的作用；
- MCP 工具调用作为 AI 能力中间层的价值；
- 数据库对会话、消息、任务和调用日志的持久化作用。

本项目中的 AI 能力采用规则和模板模拟，不依赖真实大模型 API，保证本地演示稳定可控。

## 4. 为什么使用 Web 形式

Web 系统便于课堂展示，也便于直观看到中间件链路的处理过程。前端页面可以同时展示用户问题、任务编号、任务状态、分类结果、知识库命中结果、客服回复、是否转人工以及 MCP 工具调用日志。

采用前后端分离方式还有利于说明系统职责划分：

- 前端负责交互展示和状态轮询；
- 后端负责 API、任务创建和数据持久化；
- Worker 负责异步处理；
- Redis 负责队列和缓存；
- MCP Server 负责工具能力封装。

## 5. 为什么使用 Redis

Redis 是本项目的核心中间件之一，主要承担三类职责：

1. 使用 Redis List 保存待处理任务，实现轻量级消息队列；
2. 使用 Redis Hash 保存任务状态，便于前端快速查询处理进度；
3. 使用 Redis Hash 保存会话最近状态，减少频繁查询数据库。

相比 Kafka、RabbitMQ 等消息中间件，Redis 部署简单、学习成本较低，适合本科课程大作业和本地演示。Redis List 的 `BRPOP` 命令可以实现阻塞式任务消费，满足本项目的异步处理需求。

## 6. 为什么使用 MCP

MCP（Model Context Protocol）可以将 AI 工具能力从业务系统中独立出来，让 Worker 不直接关心工具内部实现。项目中 MCP Server 提供问题分类、知识库检索、客服回复生成、工单创建和转人工判断等工具。

本项目先使用规则和模板实现 MCP 工具，保留后续扩展为真实大模型调用的接口结构。这样既降低课程项目开发复杂度，又能体现“工具调用中间层”的设计思想。

## 7. 系统最终演示目标

最终系统需要完成以下演示目标：

1. 前端创建或选择客服会话；
2. 用户输入问题并提交；
3. 后端立即返回 `task_id`；
4. Redis List 中出现待处理任务；
5. Worker 从 Redis 队列中取出任务并更新状态；
6. Worker 调用 MCP 工具完成分类、检索、回复生成和转人工判断；
7. SQLite 保存会话、消息、任务、工具调用日志和工单；
8. Redis 更新任务最新状态；
9. 前端轮询并展示任务处理结果；
10. 后台演示页可以查看任务列表和 MCP 工具调用日志。

本项目的最高优先级是跑通完整链路，其次是文档清晰、代码结构清楚和页面展示明确。
